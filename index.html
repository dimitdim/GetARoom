Project Proposal

Team Members: This group has only two team members, Dimitar Dimitrov, and Kyle Flores. Both members are in the 3:20 to 5:00 section, and will meet during that class period.

Project Summary: We plan to create a sensing network capable of determining whether or not a room is use. The system should be designed such that new sensing nodes (rooms) can be easily added to the network. Users should be able to access the system remotely and see if a room is in use, or not in use, saving users the (admittedly short) walk only to find a room occupied. To preserve the privacy of people using the rooms, only true/false data on room usage should be collected and stored. Additionally, the software will be presented in a user friendly interface.

Minimum deliverable: Software that informs the user whether a light is on in a remote room.

Extensions: There are multiple possible extensions for this project that involve improving sensing capabilities or analyzing sensor data. Additional sensors, such as temperature or sound sensors, can be added to the package. Temperature readings could detect for example, if a window has been left open in the middle of winter, leaving the room undesirable. Sound sensors can help differentiate between the presence of humans, and a forgotten light, if the most basic sensor package is assumed to be only light sensitive. After the software has existed for a long enough period of time, it can be used to predict usage patterns based on known trends. Finally, sensor data can be mapped to usage patterns to help optimize room characteristics (such as temperature) based on likelihood of use at a given time. Finally, the software interface can be made remotely accessible and be given a aesthetically pleasing interface

First step: We should start this project by developing a simple sensor package to determine if a room is in use, or not in use. This will likely involve detecting if the light is on or off (aided by the fact that most rooms at Olin have auto lights off), and if there is a high sound volume. Other instruments can later be added, but we would probably start with these.

Biggest problem: Hardware sensing and networking. One option is to use RasPi’s with the onion.io client to aid with networking

Design

0. Hardware:

  • We have an ambient light sensor and a temperature sensor attached to an Arduino
  • We will have a microphone with an envelope filter attached, time permitting
  • We also have a proximity sensor attached, which detects the opening of a door
  • We have a status LED attached as well
  • We’re using an ENC28J60 card for ethernet
  • Time permitting, we may reduce the price per kit by programming ATMega’s instead of Arduino’s and using ENC28J60 chips

1. Arduino Code

  • The Arduino code uses an ENC28J60 and an Ethercard library to host a website displaying a python dictionary of raw sensor data and hardware uptime
  • Data includes readings from the temperature and light sensors. It will also include ambient sound data.
  • Data also includes current status of door and time (in terms of hardware uptime) door last detected to be open

2. Data Collection

  • Nodes are initialised according to a configuration file; a SQL entry in the Nodes is created if the node is new
  • In the future, this file will be updated with a DHCP-assigned IP when a new sensor kit is started
  • Every 10 seconds, the current hosted website is downloaded and parsed for data values
  • A SQL entry in the Data table is created with the raw data, linking to the Node it originated from
  • Nonsensical data is disregarded

3. Data Storage

  • Data is stored and organized using Flask-SQLAlchemy

4. Data Analysis

  • A script is running detecting long-term changes in the 'last opened' value for each node
  • Every time the door is opened, the status of the room is analysed
  • If there is a significant increase and/or a rapid fluctuation of any of the parameters, the status of the room is set to occupied
  • If there is a significant decrease in any of the parameters, the status is set to vacant
  • If none of the parameters change significantly, the status does not change either
  • Statuses only change at open-door times
  • Time of change is recorded along with node and status in the Status SQL table
  • We will survey use of rooms for a period of time to validate and tune our algorithm

5. Frontend

  • A website is hosted using the Flask library for Python
  • Flask's Login Manager has users login using an OpenID and stores their information in the Users SQL table
  • Users can set preffered temperature scale - more custamization will be available in the future
  • List of Nodes with data is displayed
  • Each node link generates a page displaying the most recent Status database entry, converted for human readibility. It also diplasys the last time the data was updated
  • The page also displays the last 10 statuses, including time, by parsing the Status table
  • In the future, we will talk to IT for campus server space (ideally) or figure out a way to integrate using Heroku with the LAN-connected hardware
